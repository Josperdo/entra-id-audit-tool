# Live Mode Manual Verification Checklist

`dashboard/live.js` authenticates directly to Microsoft Graph from the browser (MSAL.js PKCE flow, via `loginRedirect`) — there is no automated test for the OAuth redirect exchange itself (see TECHNICAL.md's Testing section for why). Run this checklist against the real M365 dev tenant, the same one used to validate the PowerShell backend, whenever `live.js`, `app.js`'s `loadData` seam, or the MSAL config change.

Prerequisite: a SPA platform registered per [SETUP.md](../SETUP.md#4-spa-app-registration-setup-live-dashboard-mode), and the dashboard served locally (`npx serve dashboard` or similar) — not opened via `file://`. The registered redirect URI must exactly match the origin+port you're actually serving from.

> **Note on browser choice:** an earlier popup-based implementation was found to fail silently in Brave — the opener tab couldn't detect the popup completing even though login genuinely succeeded, because Brave blocks the cross-window polling MSAL's popup flow depends on. Live Mode now uses a full-page redirect instead, specifically to avoid that whole class of browser-specific breakage. Test in at least one Chromium-based browser (Chrome, Edge, or Brave) to catch any regression back toward that failure mode.

1. **Regression** — with no changes to Live Mode itself, confirm "Load Report (JSON)" and "Load Sample Data" still work exactly as before.
2. **First-run form** — clear `localStorage`, click **Connect Live**. The Tenant ID / Client ID form should appear; the tab should not navigate anywhere yet.
3. **Form validation** — submit with one or both fields empty; confirm an inline error appears and no MSAL call is attempted.
4. **Successful connect** — enter valid Tenant ID / Client ID, submit. The tab should navigate to a Microsoft sign-in page, followed by a consent screen listing exactly the 5 scopes (`User.Read.All`, `AuditLog.Read.All`, `Policy.Read.All`, `RoleManagement.Read.Directory`, `Application.Read.All`). After accepting, the tab lands back on the dashboard, the header shows "Connected as {upn}", and the audit **automatically resumes and runs** without a second click.
5. **Status panel** — all 6 checks should appear in the fixed order (stale-accounts, no-mfa, guest-review, privileged-roles, stale-service-principals, ca-policy-gaps) and transition queued → running → a terminal state.
6. **Licensing gap handling** — on a tenant without Azure AD Premium P1/P2, confirm `stale-accounts`, `no-mfa`, and `guest-review` show "skipped — requires Azure AD Premium P1/P2" while the other 3 complete normally with findings.
7. **Parity check** — run `.\scripts\Invoke-EntraAudit.ps1` against the same tenant and compare `privileged-roles`, `stale-service-principals`, and `ca-policy-gaps` finding counts against the Live Mode run. They must match exactly — any difference indicates the JS port has drifted from `AuditChecks.psm1`.
8. **Re-run** — click the button again (now labeled "Re-run Live Checks"). It should re-run all 6 checks silently (cached session via `acquireTokenSilent`) with **no navigation at all**.
9. **Disconnect** — click Disconnect, then Connect Live again. It should require signing in again (full redirect round-trip), not silently reuse the old session.
10. **Persistence vs. auto-connect on a plain refresh** — after connecting, just refresh the page (not via Connect Live). Tenant ID/Client ID should still be remembered, but the badge should **not** silently flip back to "Connected" and the audit must **not** re-run on its own — only an actual redirect return (step 4) or an explicit click triggers a run. A plain refresh is not a redirect return.
11. **No leaked credentials** — with DevTools open (Console + Network tabs) throughout the above, confirm no access token, ID token, or credential value ever appears in console output, in the URL bar after the redirect completes, or in the assembled report JSON (visible via "Export CSV" or by inspecting `window` state).
